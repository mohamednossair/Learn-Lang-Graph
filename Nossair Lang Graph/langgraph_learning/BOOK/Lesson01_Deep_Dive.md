## Real-World Analogy

Think of a **factory assembly line**:
- The **product** being built = the `State`
- Each **workstation** (welding, painting, packaging) = a `Node`
- The **conveyor belt** connecting stations = an `Edge`
- The factory manager never touches the product — each station does specific work and passes it on

The product (state) is always complete at every step. Each station only modifies its specific part. This is exactly how LangGraph nodes work.

---

## How It Works — Under the Hood

When you call `graph.invoke(initial_state)`, here is what LangGraph does internally:

```
1. Load initial_state into memory as the "current state"
2. Find which node START connects to (the entry node)
3. Call entry_node(current_state) → get back a dict of updates
4. For each key in the updates dict:
      current_state[key] = reducer(current_state[key], updates[key])
      (default reducer = replace the old value)
5. Check: does this node have a fixed edge?
      YES → go to that node, repeat from step 3
      NO  → check if conditional edge, call routing function
6. If next node is END → stop, return current_state
```

This means state is **one shared object**, mutated step by step. Each node call is synchronous by default (async is possible but not covered here).

## The State — TypedDict in Detail

```python
from typing import TypedDict

class OrderState(TypedDict):
    order_id:     str          # set at start
    items:        list[str]    # set at start
    total_price:  float        # calculated by pricing_node
    is_valid:     bool         # set by validation_node
    invoice:      str          # generated by invoice_node
    sent:         bool         # set by send_node
```

**Design rules for state:**
1. **Every field has a type annotation** — no bare fields
2. **Names describe what they hold**, not which node sets them
3. **Initialize ALL fields** in `initial_state` — missing keys cause `KeyError` inside nodes
4. **Use simple types**: `str`, `int`, `float`, `bool`, `list`, `dict` — avoid complex objects
5. **One field per concept** — don't pack multiple things into one string field

### Initializing state correctly

```python
# ✅ CORRECT — all keys initialized
initial = {
    "order_id":    "ORD-001",
    "items":       ["laptop", "mouse"],
    "total_price": 0.0,       # will be filled by node
    "is_valid":    False,     # will be set by node
    "invoice":     "",        # will be generated by node
    "sent":        False      # will be set by node
}

# ❌ WRONG — missing keys → KeyError when node reads them
initial = {
    "order_id": "ORD-001",
    "items":    ["laptop", "mouse"]
    # total_price, is_valid, invoice, sent MISSING
}
```

## Nodes — Everything You Need to Know

### The correct node signature

```python
def my_node(state: MyState) -> dict:
    # Read from state
    value = state["some_key"]
    # Do work
    result = transform(value)
    # Return ONLY what changed — this is merged into state
    return {"output_key": result}
```

### What a node CAN do:
- Read any state key
- Call external services (database, API, LLM)
- Do computation
- Log/print
- Raise exceptions (but better to catch and return error in state)

### What a node MUST NOT do:
- Modify state in-place: `state["x"] = value` — this mutates the dict but won't properly trigger reducers
- Return the full state: `return state` — always return only changed keys
- Return `None` — always return a dict (even `{}` if nothing changed)

### Returning nothing (no-op node)

```python
def logging_node(state: MyState) -> dict:
    """A node that only logs — doesn't change state."""
    print(f"[LOG] Current step: {state['step']}")
    return {}   # empty dict = no state changes
```

### Before vs After comparison

```python
# ❌ BEFORE (wrong — returns full state)
def process_node(state: MyState) -> dict:
    state["result"] = state["input"].upper()
    state["processed"] = True
    return state   # BUG: returns all keys including those you didn't change

# ✅ AFTER (correct — returns only changes)
def process_node(state: MyState) -> dict:
    return {
        "result":    state["input"].upper(),
        "processed": True
    }
```

## Edges — All Three Types

```python
from langgraph.graph import StateGraph, START, END

builder = StateGraph(MyState)
builder.add_node("node_a", node_a_fn)
builder.add_node("node_b", node_b_fn)
builder.add_node("node_c", node_c_fn)

# Entry edge — which node runs first
builder.add_edge(START, "node_a")

# Fixed edges — always go from A to B
builder.add_edge("node_a", "node_b")
builder.add_edge("node_b", "node_c")

# Exit edge — signal end of graph
builder.add_edge("node_c", END)
```

**START and END are sentinels**, not real nodes. They mark the boundaries of your graph.

### Multiple entry or exit points

```python
# Multiple entry nodes (parallel start — advanced)
builder.add_edge(START, "node_a")
builder.add_edge(START, "node_b")   # both run at start

# Multiple exit nodes (different end conditions)
builder.add_edge("success_node", END)
builder.add_edge("error_node", END)
```

## compile() and invoke() — What Actually Happens

### compile()

```python
graph = builder.compile()
```

At compile time, LangGraph:
1. Validates: is there a `START` entry edge?
2. Validates: is there an `END` exit path?
3. Validates: are all node names in edges actually registered?
4. Builds internal execution graph (DAG)
5. Returns a `CompiledStateGraph` object

If validation fails → `ValueError` with a clear message.

### invoke()

```python
result = graph.invoke(initial_state)           # no memory
result = graph.invoke(initial_state, config)   # with thread_id for memory
```

Returns the **final complete state dict** after all nodes have run.

### stream() — see execution in real time

```python
# stream_mode="values" — see full state after each node
for snapshot in graph.stream(initial_state, stream_mode="values"):
    print(snapshot)   # full state after each node runs

# stream_mode="updates" — see only what changed
for node_name, updates in graph.stream(initial_state, stream_mode="updates"):
    print(f"{node_name} changed: {updates}")
```

Use `stream()` during development to watch your graph execute step by step.

## Anti-Patterns — What NOT to Do

### Anti-pattern 1: Mutating state in-place

```python
# ❌ WRONG
def bad_node(state: MyState) -> dict:
    state["result"] = "processed"   # direct mutation
    return state                    # returning full state

# ✅ CORRECT
def good_node(state: MyState) -> dict:
    return {"result": "processed"}  # only return what changed
```

**Why wrong:** Direct mutation bypasses LangGraph's reducer system. When you use `add_messages` or custom reducers, mutations skip them entirely and cause subtle bugs.

### Anti-pattern 2: Storing mutable objects in state

```python
# ❌ WRONG — same list object shared across invocations
SHARED_LIST = []

def bad_node(state: MyState) -> dict:
    SHARED_LIST.append(state["item"])
    return {"items": SHARED_LIST}   # mutable reference in state

# ✅ CORRECT — always create new objects
def good_node(state: MyState) -> dict:
    return {"items": state["items"] + [state["item"]]}  # new list
```

### Anti-pattern 3: Heavy computation in routing functions

```python
# ❌ WRONG — routing function calls LLM (slow, stateful)
def route(state):
    resp = llm.invoke([HumanMessage("classify: " + state["text"])])
    return resp.content   # routing should be fast, pure

# ✅ CORRECT — classification done in a node, routing reads result
def classify_node(state) -> dict:
    resp = llm.invoke([HumanMessage("classify: " + state["text"])])
    return {"category": resp.content}

def route(state) -> str:
    return state["category"]   # pure read, instant
```

## Debugging Guide

### Step 1: Add print statements to every node

```python
def my_node(state: MyState) -> dict:
    print(f"[my_node] INPUT: {state}")
    result = do_work(state)
    print(f"[my_node] OUTPUT: {result}")
    return result
```

### Step 2: Use stream() to watch execution

```python
for step in graph.stream(initial_state, stream_mode="updates"):
    node_name = list(step.keys())[0]
    updates   = list(step.values())[0]
    print(f"✓ {node_name} → {updates}")
```

### Step 3: Visualize the graph

```python
print(graph.get_graph().draw_mermaid())
# Paste output at https://mermaid.live
```

### Common error messages

| Error | Cause | Fix |
|-------|-------|-----|
| `ValueError: Graph must have an entrypoint` | Missing `add_edge(START, ...)` | Add entry edge |
| `AttributeError: 'StateGraph' has no 'invoke'` | Called on builder, not compiled graph | `graph = builder.compile()` first |
| `KeyError: 'field_name'` inside node | Field not in initial state | Add it to initial state dict |
| `InvalidUpdateError` | Node returned invalid updates | Return only valid state keys |
| Graph hangs / never returns | No `add_edge(..., END)` | Add exit edge |

## Best Practices Checklist

```
STATE DESIGN:
  □ Use TypedDict, never plain dict
  □ Every field has a type annotation
  □ All state keys initialized (even to empty "" / 0 / [] / False)
  □ Field names describe their content, not which node fills them

NODE DESIGN:
  □ Nodes return only changed keys (never full state)
  □ Nodes have type hints: (state: MyState) -> dict
  □ Nodes handle exceptions internally (never let them crash the graph)
  □ Pure nodes return {} if they have no state changes (e.g., logging)

GRAPH STRUCTURE:
  □ add_edge(START, "first_node") — always present
  □ add_edge("last_node", END) — always present
  □ builder.compile() before any invoke() call
  □ Initial state passed to invoke() contains all TypedDict keys

DEBUGGING:
  □ Use stream_mode="updates" to watch step-by-step execution
  □ Print graph.get_graph().draw_mermaid() to visualize structure
  □ Add print("[node_name] state=...") at start of each node during dev
```

## Interview Q&A — Lesson 1

**Q1: What is the difference between `invoke()` and `stream()`?**

`invoke()` runs the complete graph and blocks until all nodes have finished, returning the final state as a single dict. `stream()` runs the graph but yields intermediate results as each node completes — the caller sees progress in real time. Use `invoke()` for batch processing or when you only care about the final result. Use `stream()` for: chat UIs (stream tokens), monitoring dashboards (watch node execution), debugging (see what each node changed), and long-running agents where you want to show progress. The `stream_mode` parameter controls what you receive: `"values"` = full state snapshot, `"updates"` = only changed keys, `"messages"` = LLM token chunks.

**Q2: Why do nodes return only the keys they changed, not the full state?**

This is the "partial update pattern" and it has three critical benefits: (1) **Reducer safety** — returning partial updates allows LangGraph's reducers to work correctly. If you returned the full state with `messages: [all_messages]`, you'd bypass the `add_messages` reducer. (2) **Isolation** — each node declares exactly what it owns. A `pricing_node` returning `{"total": 99.0}` can't accidentally overwrite `{"invoice": "already generated"}` that a later node set. (3) **Testability** — you can test `pricing_node` in isolation by checking `result["total"]` without needing the full state context.

**Q3: What happens if two nodes run in parallel and both write to the same state key?**

If the field has no reducer (default), LangGraph raises a conflict error — it can't determine which value wins. Solution: always define a reducer for fields that parallel nodes can write to. For lists: use `add_messages` or a custom append function. For numeric counters: `lambda x, y: x + y`. For the most recent value: `lambda old, new: new`. LangGraph enforces this at graph compile time when it detects parallel paths writing to the same key.

**Q4: TypedDict vs Pydantic BaseModel for state — which is better and why?**

`TypedDict` is strongly preferred for graph state. Reasons: (1) Zero runtime validation overhead — state is updated after every node execution; Pydantic would validate on every update adding latency. (2) Works naturally with Python's type system — no extra class machinery. (3) LangGraph's internal implementation is optimized for TypedDict. (4) Simpler — no `model_config`, `validators`, or `@field_validator` needed. Use Pydantic for: tool input schemas (where you want to validate user-provided data), API request/response models (Lesson 15), structured LLM outputs. Never use Pydantic as the graph state type.

**Q5: What does `compile()` actually do and can you skip it?**

No, you cannot skip it. `compile()` does: (1) validates graph structure (entry point, exit path, all referenced nodes defined), (2) resolves all conditional edges into an internal DAG representation, (3) sets up the checkpointer connection if provided, (4) returns a `CompiledStateGraph` which is the object you actually `invoke()` on. The `StateGraph` builder has no `invoke()` method — calling it on the builder raises `AttributeError`.

**Q6: What is `get_graph().draw_mermaid()` and how do you use it professionally?**

`graph.get_graph().draw_mermaid()` generates a Mermaid diagram string of your entire graph — all nodes as boxes, all edges as arrows, with conditional edges labeled. Uses: (1) visualize and verify graph structure before deploying, (2) generate documentation for team members (paste in README.md), (3) debug routing — visually confirm all paths exist. In Jupyter: `display(Image(graph.get_graph().draw_mermaid_png()))`. In CI: generate and compare diagrams to detect unintended structural changes.

**Q7: How do you add optional state fields that only some nodes use?**

Two approaches: (1) Include in TypedDict with `Optional` type and initialize to `None`:
```python
class MyState(TypedDict):
    required_field: str
    optional_field: Optional[str]   # None until set
```
(2) Use `total=False` on TypedDict for completely optional keys:
```python
class MyState(TypedDict, total=False):
    optional_key: str   # nodes can check: "optional_key" in state
```
Approach 1 is preferred — always initialize to `None` rather than leaving keys missing. Nodes then check `if state["optional_field"] is not None:` before using the value.
